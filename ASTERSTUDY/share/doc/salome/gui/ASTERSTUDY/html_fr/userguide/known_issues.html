Problèmes connus¶

Cette page répertorie les problèmes connus non encore résolus au moment de la livraison de cette version.

Dans la gestion de l’étude¶

Utilisation de caractères spéciaux dans les chemins de fichiers¶

Il est déconseillé d’utiliser des caractères spéciaux (espaces, caractères accentués, caractères non ASCII comme @, !, /, #, %, $ etc…) dans les chemins de fichiers, que ce soit le nom de l’étude (fichier .hdf) comme les fichiers d’entrée et de sortie. Cela peut en effet induire des difficultés :

  • à lancer les calculs (espaces) ;
  • à sauvegarder l’étude (accents) ;
  • à ouvrir des fichiers de résultats dans le module ParaViS (accents).

L’application avertit l’utilisateur en cas d’utilisation de caractères spéciaux.

Chargement d’une étude¶

Quand on recharge une étude réalisée auparavant (Fichier > Ouvrir, fichier .hdf), l’interface ne bascule pas immédiatement vers AsterStudy. Il suffit de cliquer sur l’icône.

Dans la mise en données¶

POST_RELEVE_T command¶

Lors de l’édition de la commande POST_RELEVE_T, le lien au maillage n’est pas actif tant qu’on n’a pas validé les modifications du mot-clé facteur ACTION ou la commande entière.

Commande INCLUDE_MATERIAU¶

Le mot-clé FICHIER accepte un nom de fichier à inclure en tant que chaine de caractères. L’application ne sait rien à propos de ce fichier.

Contournement : Insérer une étape en mode texte en dessus de l’étape qui utilise cette commande INCLUDE_MATERIAU et déclarer le fichier en entrée, choisir un numéro d’unité logique libre. Ensuite, modifier la valeur de FICHIER en fort.NN avec le numéro d’unité choisi.

Mise en donnée d’une commande¶

Pour une meilleure expérience utilisateur (visualisation intégrée du maillage), il est nécessaire de renseigner les mots-clés obligatoires du premier niveau d’édition avant d’éditer un mot-clé possédant un second niveau d’édition.

Attention, ces mots-clés obligatoires peuvent parfois être tout en bas du panneau d’édition (par exemple le mot-clé MODELE dans la commande AFFE_CHAR_MECA).

Performance de la vue maillage¶

Les commandes relatives aux maillages affichent un maillage avec les groupes sélectionnés dans la vue centrale de l’application. Pour des maillages de grande taille, avec beaucoup de groupes, l’affichage du maillage peut prendre beaucoup de temps.

Contournement : il est possible de désactiver la visualisation de ces opérations en désactivant la vue maillage. Ceci peut se faire en désactivant le bouton Afficher la vue maillage de la barre d’outils.

Dans la gestion des fichiers¶

Ajout de fichiers dans le panneau Fichiers¶

Lorsque l’on ajoute des fichiers en mode texte, il est nécessaire de bien penser à valider par OK (panneau de droite) après avoir choisi le fichier d’entrée ou de sortie dans la boite de dialogue.

Ajout de fichiers distants¶

Pour les utilisateurs EDF, les systèmes de fichiers des serveurs EOLE et ATHOS sont automatiquement montés lors de l’ouverture du module. Pour accéder rapidement à votre espace personnel, il est nécessaire de sélectionner dans la barre latérale le serveur désiré et de taper « home/NNI » ou « scratch/NNI » dans le champ de saisie textuelle.

Dans le lancement des calculs¶

Utilisation de l’option Bases distantes¶

Lors du lancement d’un calcul avec l’option Bases distantes (remplie ou juste sélectionnée), le fichier log dans la liste déroulante »Voir le fichier des messages » n’est plus disponible (seuls les fichiers « message » et « export » sont disponibles). Si l’option est remplie et non sélectionnée, on observe le même phénomène. Ce paramètre ne doit pas être modifié une fois renseigné, sa modification entrainera alors une erreur fatale.

Autres problèmes¶

En cas d’erreur de l’application - Mode de récupération¶

Dans le cas où l’application rencontre une erreur non prévue, une boîte de dialogue Unexpected error s’affiche. Il est alors possible que l’enregistrement de l’étude (fichier .hdf) se passe mal. Dans un tel cas, préférez utiliser Fichier > Enregistrer sous pour ne pas effacer le contenu précédent.

Quand l’étude est ensuite relue, elle peut être vide dans AsterStudy.

Mode de récupération : Si l’étude ne peut pas être restaurée depuis le fichier .hdf, on essaie d’utiliser automatiquement le mode de récupération pour restaurer le Cas Courant avec ses Étapes.

À chaque fois que l’étude est enregistrée, un fichier supplémentaire est écrit dans le répertoire xxx_Files de l’étude sous le nom data.ajs. Si le fichier .hdf est corrompu, vous pouvez utiliser le Mode de récupération depuis le menu Opérations > Mode de récupération. Sélectionnez simplement le fichier data.ajs à récupérer.

Autre solution de contournement : dans l’onglet History view, aller dans le menu Opérations > Importer un Cas, sélectionner le fichier backup.export dans le Dossier personnel ($HOME), puis .asterstudy/backup. Ce dossier correspond toujours à la dernière sauvegarde ; le dossier backup.1 à la sauvegarde précédente, et ainsi de suite. Il faut donc que l’étude ait été sauvegardée régulièrement pour récupérer cet état. Par défaut, les répertoires commençant par un point sont cachés. Si c’est le cas, il faut taper .asterstudy dans la zone de texte (si la zone n’est pas visible, taper Ctrl+L pour l’afficher).

Blocage possible de l’écran de déverrouillage¶

Pour les utilisateurs EDF, le changement de connexion réseau peut entrainer des difficultés à déverrouiller le poste lorsque salome_meca est en marche. Il est conseillé de sauvegarder et fermer salome_meca lors d’un changement de connexion réseau.

Contournement : si un utilisateur rencontre ce problème, il est possible de déverrouiller l’écran après avoir tué manuellement le processus “gvfsd”. Cette action est réalisable en mode console (CTRL+ALT+F1) et en utilisant la commande “pkill gvfsd”.
